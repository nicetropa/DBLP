1 Introduction
Le projet du cours INFO-F-203 Algorithmique 2 (année académique 2025–2026) porte sur l’ana-
lyse de la base de données DBLP [ 1], qui est une base de données de publications scientifiques
en informatique. L’objectif principal est de mettre en pratique les notions vues au cours jusqu’à
la section 7 composantes fortement connexes .
Le projet doit se faire en binômes (2 étudiant·e·s). Une fois votre binôme constitué, formez
un groupe de deux sur l’UV : https://uv.ulb.ac.be/mod/choicegroup/view.php?id=
1593747 . La liste des personnes n’ayant pas encore de binôme est disponible sur cette page,
dans le groupe « Personnes n’étant pas encore dans un binôme ». Si vous n’avez pas encore de
binôme, contactez les personnes de cette liste.
L’examen en première session inclut une question sur le projet. Le projet compte en bonus pour
1/4 de la note du cours, donc uniquement si la note est supérieure à celle de l’examen et si la
question d’examen sur le projet est réussie. En résumé :
if question_examen_projet_reussie and note_projet > note_examen :
note_finale =
3/4 * note_examen + 1/4 * note_projet
else:
note_finale = note_examen
où la note de l’examen inclut la note de la question de l’examen sur le projet.
Votre travail doit être rendu sur l’UV au plus tard le 26 avril 2026 à 23h59 . En cas de retard,
les pénalités suivantes seront appliquées sur la note du projet :
—moins de 2 heures de retard : -2/20 ;
—entre 2 et 24 heures de retard : -5/20 ;
—plus de 24 heures de retard : la note du projet devient 0/20.
Si des cas de force majeure vous empêchent de rendre votre travail à temps, veuillez me contacter
(voir section 7). Une seule personne du binôme doit remettre le projet sur l’UV. Le nom des deux
personnes ainsi que le numéro de matricule doivent apparaître dans le rapport.
La pondération pour la note du projet est de 60% pour le rapport et 40% pour le code. Les détails
sur les délivrables et les critères attendus sont donnés en section 4.
L’objectif de ce projet est d’exploiter un flux de publications DBLP afin d’identifier des “commu-
nautés” de chercheur·euse·s selon deux points de vue complémentaires :
—Communautés de co-publication (graphe non orienté) : des auteur·rice·s sont regroupé·e·s
s’il existe une chaîne de co-publications les reliant. Plus de détails sont fournis dans la
section 3.2.

2 Ce qui est mis à votre disposition
Téléchargez le fichier XML compressé du snapshot du premier janvier 2026 de DBLP https://
drops.dagstuhl.de/storage/artifacts/dblp/xml/2026/dblp-2026-01-01.xml.gz [2].
Il n’est pas nécessaire de décompresser ce fichier : le code fourni lit directement le format
.xml.gz .
Vous recevrez sur l’UV les éléments suivants :
—un fichier dblp.dtd (DTD officielle associée) [ 3],
—deux fichiers Java fournis afin que vous n’ayez pas à implémenter un parseur XML et afin
d’illustrer l’interface d’utilisation :
—DblpPublicationGenerator.java ,
—DblpParsingDemo.java .
Le code Java peut être compilé avec la commande suivante :
javac DblpPublicationGenerator.java DblpParsingDemo.java
Le code peut ensuite être exécuté avec la commande suivante :
java DblpParsingDemo dblp-2026-01-01.xml.gz dblp.dtd
Vous pouvez également ajouter l’argument --limit=N pour vous limiter aux premières N publi-
cations du fichier xml.
Le parseur fourni vous donne, publication par publication, un objet contenant la liste ordonnée
des auteur·rice·s de la publication. Le principe est illustré dans la démo DblpParsingDemo :
vous itérez tant que nextPublication() renvoie une publication, puis vous récupérez la liste
authors . L’auteur·rice en première position est authors.get(0) ; les autres auteur·rice·s (s’il y
en a) sont authors.get(1) ,authors.get(2) , …
Types de publications considérés. Le générateur fourni DblpPublicationGenerator ne pro-
duit que les publications de type article ,inproceedings ,book etincollection . Les autres
types présents dans DBLP sont automatiquement ignorés. Ce choix est volontaire pour le projet :
ne modifiez pas le générateur pour inclure d’autres types de publications.
Non nécessaire pour le projet : pour plus d’information sur la structure de DBLP et les fichiers
XML et DTD, vous pouvez consulter [ 4] qui est disponible sur https://dblp.org/xml/docu/
dblpxml.pdf .

votre programme dispose déjà d’un état interne cohérent permettant de répondre correctement
aux questions demandées sur ces kpublications, sans relire les publications précédentes.
Autrement dit, votre programme ne peut pas attendre la fin du flux pour commencer à construire

3.2 Tâche 1
3.2.1 Modèle
On considère un graphe non orienté des auteur·rice·s :
—un sommet = un auteur·rice (nom tel que fourni par DBLP),
—une arête entre AetBs’il existe au moins une publication où AetBapparaissent tous les
deux (co-auteur·rice·s).
Une communauté au sens de cette tâche correspond à un ensemble d’auteur·rice·s relié par co-
publication (directe ou via une chaîne). En d’autres termes, AetBappartiennent à la même
communauté si et seulement si il existe un chemin entre AetB.
3.2.2 Résultat attendu
En lisant les publications une par une, vous devez maintenir ces communautés online . Concrète-
ment :
—à chaque nouvelle publication, vous mettez à jour les communautés en conséquence (tous
les co-auteur·rice·s d’une même publication doivent être regroupé·e·s),
—vous devez maintenir en permanence la taille de chaque communauté (nombre de
membres), de façon online également.
3.2.3 Sorties demandées
Votre programme produira au minimum :
1.A la fin du parsing, un histogramme des tailles des communautés dans un fichier de sortie.
2.Exigence online à démontrer : vous afficherez aussi un état intermédiaire sur le terminal,
toutes les N publications lues (par exemple, N= 105), le nombre de communautés et la
taille des 10 plus grandes communautés.
3.3 Tâche 2
3.3.1 Construction du graphe orienté
Pour chaque publication ayant au moins 2 auteur·rice·s (dans l’ordre donné dans la liste ordon-
née de DBLP) :
—soitAla personne en première position,
—pour chaque autre auteur·rice Bde la publication ( B̸=A), on incrémente un compteur
pour la paire ordonnée (A→B).
Après lecture de toutes les publications, on construit un graphe orienté où l’on conserve une arête
A→Bsi et seulement si le compteur associé est ≥6.
Remarque : le seuil « 6 » vise à éviter que tout se regroupe en composantes énormes et à faire émerger
des communautés plus significatives.

3.3.2 Communautés recherchées
Dans ce graphe orienté filtré, on s’intéresse à des communautés au sens suivant :
Un groupe d’auteur·rice·s forme une communauté si et seulement si, pour toute paire
(u, v)dans le groupe, il existe un chemin orienté de uversvetdevversu.
3.3.3 Diamètre des 10 plus grandes communautés
Pour chacune des 10 plus grandes communautés, vous devez calculer son diamètre dans le graphe
orienté filtré. La définition dans une communauté donnée Cest :
max u∈C, v∈Cd(u, v)
où la distance d(u, v)est la longueur (nombre d’arêtes) du plus court chemin orienté deuvers
v,en restant dans C. Autrement dit, le diamètre doit être calculé dans le sous-graphe induit par
la communauté, pas dans le graphe global avec chemins qui sortent/reviennent.
3.3.4 Sorties demandées
Votre programme écrira au minimum :
—un histogramme des tailles de communautés ;
—les 10 plus grandes communautés : pour chacune,
—sa taille,
—son diamètre,
—les noms des auteur·rice·s de la communauté.
Pour rappel, pour cette tâche, toutes les notions (chemins, communautés, diamètre) sont calculées
dans le graphe orienté après filtrage (seuil ≥ 6 publications).
3.4 Tâche bonus
Bonus (+1 point sur 20) si vous implémentez cette partie suivante correctement. Pour cette
partie-ci, vous pouvez l’implémenter en Python (ou en Java si vous préférez) en utilisant les sorties
de votre programme Java.
Pour les 10 plus grandes communautés de la tâche B, déterminez si elles sont associées à un ou
plusieurs pays .
Pour cela, vous pouvez utiliser l’API OpenAlex [5] pour faire des requêtes pour identifier l’ins-
titution dans laquelle travaille chaque auteur·rice. La documentation de l’API est disponible à
l’adresse https://developers.openalex.org/api-reference/introduction
Il vous est ensuite demandé de produire un graphique indiquant la proportion d’auteur·rice·s
travaillant dans différents pays au sein de chaque communauté.
